# Refinements — 2026-06-25

First full end-to-end run of the SDD flow on the Packet Config / Add Team project.

---

## What worked

- Linear issue creation from `/spec-author` when no issues existed — issues were created and specs generated from them.
- `hops linear --edit=false` correctly linked the commit to the Linear issue and opened the PR with the right title.
- `/spec-builder` built the route shell correctly from the spec, including rollout integration and entry-point updates.
- `/review-pr --auto-comment` posted inline findings to the draft PR without any manual intervention.
- Draft PR was kept open — never marked ready for review.

## What didn't work / was slow

- **Spec audit loop took too long.** 3 rounds × multiple issues = significant wait. Critical findings surfaced as late as round 3, which means the first-pass spec quality was not good enough. Fix: reduced MAX_ROUNDS to 2 and improved round summary logging.
- **Spec author first pass had weak grounding.** Several `file:line` citations were wrong or invented on the first pass, forcing the audit loop to correct them. Fix: spec-author needs to verify citations more aggressively before writing.
- **Several setup errors before spec audit ran the first time.** The workflow had bugs that needed patching before it would execute cleanly.
- **Rollout/experiment logic was specced incorrectly.** The agent picked `%i[backbone]` instead of `%i[redux_rollout]` because the rollout framework wasn't documented in the spec context. Fix: document rollout/experiment framework as a required input before spec authoring.
- **PR grew bigger than expected after `/fe-review` applied fixes.** The issue touched ~10 files in the spec, but after review fixes the PR was larger. The 10-file cap needs to account for test files and review-driven additions, not just the original build files.

## Changes made to the process

- `spec-review-loop.js` renamed to `spec-audit-loop.js` — naming now consistent everywhere.
- `MAX_ROUNDS` reduced from 3 to 2 in `spec-audit-loop.js` and `spec-audit/SKILL.md`.
- Round summary logging added to `spec-audit-loop.js`: after each rejected round, the blocking categories and reviewer summary are logged so the engineer can see at a glance why it failed.
- "Always draft" rule hardened: added to `spec-builder` hard rules, `HANDOFF.md`, `AGENTS.md`, and removed the "Ready for review" line from `fe-review` wrap-up.
- All "Spec Review Complete" headers changed to "Spec Audit Complete" for consistency.

## Open items / next to tackle

- Linear issue template: when creating issues, follow the HRM template (description, AC, dev notes, dependencies, linked issues in order).
- Flow guidance: each skill should end with a clear prompt/question guiding to the next step.
- Rollout/experiment step: add a dedicated question in `/setup` or `/spec-author` asking whether a rollout is needed, and document the Homebase rollout framework inline.
- Branch setup automation: at start of `/spec-builder`, auto-detect if on main and guide to feature branch creation.
- Cost optimization: assign models per task (Haiku for cheap lookups, Sonnet for authoring, Opus for adversarial review).
